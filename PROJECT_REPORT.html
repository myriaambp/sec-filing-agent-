1. Project Overview

FilingLens is a multi-agent AI system that acts as an institutional-grade SEC filing analyst. A user asks a natural language question about any public company (e.g., "Should we be long or short on Nvidia going into next quarter?"), and the system:

  1. Collects real SEC filings from EDGAR and stock price data from Yahoo Finance at runtime
  2. Explores the data by analyzing management language shifts, computing market signal correlations, and comparing against competitors — all via parallel AI agent execution
  3. Hypothesizes by generating a structured Buy/Sell/Hold recommendation with a data-driven confidence score, a research memo, a visualization, and downloadable data artifacts

The system uses Google ADK (Agent Development Kit) as the agent framework with Gemini 2.0 Flash via Vertex AI as the LLM. Four distinct agents coordinate through an orchestrator using a parallel fan-out pattern. The frontend is a React application with a dark financial terminal aesthetic, deployed on GCP Cloud Run.

Key Differentiator: Unlike generic LLM chatbots, FilingLens never answers from model weights. Every claim is grounded in real data fetched at runtime — specific uncertainty scores, filing dates, return percentages, and word counts that the user can verify via linked EDGAR filings and downloadable CSVs.

3. Step 1: Data Collection

3.1 SEC EDGAR API

File: backend/tools/fetch_filings.py

The system fetches real SEC filings from the EDGAR database at runtime. No data is pre-cached or hard-coded.

Data Flow

  1. get_cik_for_ticker(ticker) — Resolves a stock ticker (e.g., "NVDA") to a SEC CIK number by fetching https://www.sec.gov/files/company_tickers.json
  2. fetch_recent_filings(ticker, form_type, num_filings) — Fetches filing metadata from https://data.sec.gov/submissions/CIK{cik}.json, returning accession numbers, filing dates, periods, and primary document names
  3. fetch_filing_text(accession_number, cik, primary_doc) — Downloads the actual filing HTML from https://www.sec.gov/Archives/edgar/data/{cik}/{accession}/{doc}, strips HTML with BeautifulSoup, and extracts the MD&A and Risk Factors sections using regex pattern matching

Section Extraction

The _extract_key_sections() function identifies section boundaries in the filing text using regex patterns. It searches for:

  • Risk Factors (Item 1A in 10-K, Item 1A in 10-Q): patterns like item\s+1a[\.\s\-\:]+risk\s+factors
  • MD&A (Item 7 in 10-K, Item 2 in 10-Q): patterns like management.s\s+discussion\s+and\s+analysis

It skips table-of-contents matches by iterating from the last occurrence backward and requiring the extracted section to be at least 500 characters.

Data Volume

Per query: 18 filings fetched (6 per company × 3 companies), totaling ~500,000 characters (~125,000 tokens) of extracted filing text. The full unextracted filings total ~3,000,000 characters (~758,000 tokens) — far too large to fit in any LLM's context window. This satisfies the "non-trivial dataset" requirement.

3.2 Yahoo Finance API

File: backend/tools/fetch_prices.py

Stock price data is fetched via the yfinance Python library. For each company analyzed (primary + competitors + S&P 500 benchmark), the system fetches 2-3 years of daily OHLCV data.

Computations Over Price Data

FunctionWhat It ComputesData Used
compute_returns_around_date()30-day and 60-day stock returns after each filing date, benchmarked against SPYClose prices around filing dates
_compute_volatility()Annualized realized volatility (std of daily returns × √252) for 30 days pre and post filingAll daily returns in 60-day window
_compute_volume_spike()Whether trading volume spiked on filing date vs 20-day averageVolume data for 30-day window
compute_price_summary()Total return, annualized volatility, max drawdown, Sharpe ratio, avg/max volumeEntire 500+ day price history

Data Volume

Per ticker: 500+ trading days × 6 columns (Date, Open, High, Low, Close, Volume) = 3,000+ data points. For a typical query with 3 companies + SPY: ~12,000+ raw price data points, plus derived statistics computed across the full history.

4. Step 2: Exploratory Data Analysis

EDA is performed by two specialized agents running in parallel via asyncio.gather().

4.1 Filing NLP Agent

File: backend/agent_definitions/filing_nlp_agent.py
Tool: fetch_and_analyze_filings(ticker, form_type, num_quarters)

This agent calls its tool, which internally invokes the EDGAR fetcher and NLP analyzer. The tool returns a JSON object with quarterly language scores. The agent then formats this into a structured response.

NLP Analysis Pipeline

For each filing's extracted text, the system computes (see Section 10 for full methodology):

  • Uncertainty score (0–1): Ratio of hedging/risk words scaled by 40×
  • Sentiment score (-1 to +1): Balance of confidence vs uncertainty words scaled by 50×
  • Key risk phrases: Top 5 sentences by uncertainty word density, filtered for boilerplate
  • Trend direction: "improving" (<-10% change), "deteriorating" (>+10%), or "stable"
  • Trend magnitude: Percentage change in uncertainty score vs prior quarter

4.2 Market Signal Agent

File: backend/agent_definitions/market_agent.py
Tool: compute_market_signal(ticker, filing_dates, uncertainty_scores)

This agent receives the filing dates and uncertainty scores from the FilingNLP Agent's output (passed through the orchestrator) and computes market correlations.

Signal Computation Pipeline

  1. correlate_language_with_returns() — Joins each quarter's uncertainty score with the stock's 30/60-day return after that filing date
  2. compute_signal_strength() — Splits quarters into above/below median uncertainty, computes average returns for each group, and checks how often high uncertainty predicted underperformance vs S&P 500

Signal strength classification:

StrengthCriteria
StrongReturn spread > 5% AND accuracy > 60%
ModerateReturn spread > 2% OR accuracy > 50%
WeakNeither condition met

4.3 Parallel Execution

File: backend/agent_definitions/orchestrator.py, line ~188

The orchestrator runs the MarketSignalAgent and all competitor FilingNLPAgent instances simultaneously:

tasks = [_run_agent(market_agent, market_input)]
for comp in competitor_tickers:
    tasks.append(_run_agent(filing_nlp_agent, comp_input))
results = await asyncio.gather(*tasks, return_exceptions=True)

For a query about NVDA with AMD and INTC as competitors, three agent runs execute concurrently:

  1. MarketSignalAgent analyzing NVDA stock returns
  2. FilingNLPAgent analyzing AMD filings
  3. FilingNLPAgent analyzing INTC filings

5. Step 3: Hypothesis Formation

5.1 Recommendation Engine

The recommendation is determined by a rule-based engine (not the LLM) that combines language trend and market signal:

DeterioratingStrong/ModerateNoUNDERWEIGHT (Cautionary)
ImprovingStrong/ModerateYesBUY (Bullish)
ImprovingStrong/ModerateNoOVERWEIGHT (Bullish)
Stable / Weak signalAnyAnyHOLD (Cautionary)

5.2 Confidence Score

See Section 11 for full methodology. Confidence is computed from 6 data-driven factors with a base of 30%, ranging from 15% to 95%.

5.3 Research Memo

The Analyst Agent writes a structured Goldman Sachs-style research memo with these sections:

  • Investment Thesis — Core argument in 2-3 sentences
  • Language Trend Analysis — Quarter-by-quarter breakdown with specific scores
  • Market Signal Analysis — Historical return data and S&P 500 comparison
  • Competitive Landscape — Peer filing language comparison
  • Risk Factors — What could invalidate the recommendation
  • Conclusion — Restatement with the single most important supporting data point

5.4 Visualization

File: backend/utils/chart_generator.py

A dual-axis matplotlib chart is generated showing:

  • Left Y-axis: Uncertainty score over time (red line with markers)
  • Right Y-axis: Stock price over time (green line)
  • Competitor uncertainty as dashed lines in different colors
  • Filing dates marked with vertical dotted lines
  • Dark terminal aesthetic (#0f1117 background) matching the frontend

The chart is returned as a base64-encoded PNG and displayed inline in the frontend.

9. Agent Design & Prompts

9.1 TickerParser Agent

Purpose: Extract stock ticker symbols and form type from natural language questions.

Output: {"primary_tickers": ["NVDA"], "competitor_tickers": ["AMD"], "form_type": "10-Q"}

If no competitors are specified, the orchestrator auto-detects them from a sector peers lookup table covering 20+ major tickers.

9.2 FilingNLP Agent

Purpose: Fetch and analyze SEC filing language for a company.

Key prompt instruction: "You MUST call the fetch_and_analyze_filings tool first. Do NOT generate data from your own knowledge."

This ensures the agent always uses real EDGAR data rather than hallucinating filing content.

9.3 MarketSignal Agent

Purpose: Correlate filing language uncertainty with subsequent stock performance.

Key prompt instruction: "You MUST call the compute_market_signal tool first. Do NOT generate data from your own knowledge."

9.4 Analyst Agent

Purpose: Synthesize all findings into a structured research memo.

Output structure: The prompt specifies exact JSON schema and instructs the memo to follow Goldman Sachs-style formatting with Investment Thesis, Language Trend Analysis, Market Signal Analysis, Competitive Landscape, Risk Factors, and Conclusion sections.

Design Decision: The orchestrator overrides the LLM's recommendation, confidence, language trend, and chart with deterministically computed values. This prevents the LLM from hallucinating confidence numbers or generating fake chart data. The LLM only writes the qualitative memo text.

10. NLP Methodology

File: backend/tools/analyze_language.py

10.1 Word Lists

Two curated sets of financially-relevant words:

CategoryCountExamples
Uncertainty24 wordsmay, might, could, risk, adverse, decline, volatility, litigation, deterioration, susceptible
Confidence24 wordsstrong, growth, exceeded, surpassed, delivered, achieved, profitable, efficient, demand, contributed

10.2 Scoring

  • Uncertainty Score: min(1.0, uncertainty_word_count / total_words × 40) — Typical SEC filing ratios are 0.005–0.025, so the 40× scaling maps this to a 0.2–1.0 range for meaningful differentiation.
  • Sentiment Score: clamp(-1, 1, (confidence_count - uncertainty_count) / total_words × 50) — Positive means net confident, negative means net uncertain.

10.3 Risk Phrase Extraction

Sentences are scored by uncertainty word density (uncertainty_words / total_words_in_sentence). The top 5 are returned, filtered through boilerplate patterns that skip legal disclaimers like "forward-looking statements can also be identified by words such as..."

10.4 Trend Detection

Compares the most recent quarter's uncertainty score to the prior quarter:

  • >+10% change → deteriorating
  • <-10% change → improving
  • Otherwise → stable

11. Confidence Score Methodology

File: backend/tools/compute_signal.py — generate_recommendation()

Confidence is computed from 6 independent factors, each contributing a bounded amount:

FactorMaxHow Computed
Base30%Starting confidence for any recommendation
Signal strength+20%Strong: +20%, Moderate: +12%, Weak: +4%
Trend magnitude+15%min(0.15, |trend_magnitude| / 200) — bigger language shift = more confident
Filing count+10%min(0.10, n_filings × 0.02) — more data = more confident
Trend consistency+10%+10% if scores are monotonically trending, +5% if ≥70% consistent
Return spread+10%min(0.10, |avg_high_unc_return - avg_low_unc_return|)
Competitor divergence+5%+5% if primary company's trend is unique among peers

Range: 15% (minimum clamp) to 95% (maximum clamp)

Example: NVDA with a moderate signal (+12%), 16% trend magnitude (+1.2%), 6 filings (+12%), inconsistent trend (+0%), 4.9% return spread (+4.9%), no competitor divergence (+0%) = 30% + 12% + 1.2% + 12% + 0% + 4.9% + 0% = 60.1% confidence.

Error Handling

  • No filings found: Returns error message instead of hallucinated data
  • Agent doesn't call tool: Strengthened prompts + orchestrator overrides ensure data-grounded output
  • JSON parse failures: Fallback handling strips markdown code fences, handles malformed LLM output
  • Market data unavailable: Graceful degradation — analysis continues without market signal
  • Numpy bool serialization: Explicitly cast to Python bool() for JSON compatibility

15. Limitations & Future Work

Current Limitations

  • NLP is rule-based: Word list approach is simple but effective. A financial sentiment model (e.g., FinBERT) would capture more nuanced language.
  • Section extraction is regex-based: Some filings have non-standard HTML structure, causing partial or missed section extraction.
  • Sentiment skews negative: SEC filings are legally cautious by nature, so sentiment is almost always negative. The trend (change over time) is more informative than the absolute value.
  • EDGAR rate limiting: SEC limits to 10 requests/second. Each analysis takes 30-60 seconds due to sequential filing fetches.
  • No session memory: Each query is independent — no follow-up questions or iterative refinement.

Future Improvements

  • Caching: Redis cache for EDGAR and yfinance calls to make repeat queries instant
  • FinBERT: Use a pre-trained financial sentiment model for more accurate language analysis
  • Iterative refinement: Let users ask follow-up questions with session context
  • More filing sections: Analyze forward-looking statements, financial tables, and footnotes
  • Real-time SSE: True server-sent events for smoother streaming UX
